On-chain filled-amount sources (research notes, retained as reference)
----------------------------------------------------------------------
**Common pattern (CoW + 1inch):** protocols overwrite the fill slot on cancel/invalidate, so
the adapter MUST read the fill amount *before* issuing the on-chain cancel. BoringSwapper
already does this in `_cancelOrder` — reads `filledAmount()` at L235, then runs `cancelLimitOrder`
at L250+ which calls the protocol's invalidate.

CoW
  Contract:  GPv2Settlement @ 0x9008D19f58AAbD9eD0D60971565AA8510560ab41
  Read:      filledAmount(bytes orderUid) -> uint256
  Slot:      2 (verified empirically via the CoW test)
  Units:     sell-token units (adapter restricted to sell orders)
  Semantics: monotonic accumulator, capped at order.sellAmount
  Gotcha:    invalidateOrder(uid) sets slot to type(uint256).max.

1inch v4
  Contract:  AggregationRouterV6 @ 0x111111125421cA6dc452d289314280a0f8842A65
             (standalone LOP v4 also at 0x119c71D3BbAC22029622cbaEc24854d3D32D2828)
  Read:      rawRemainingInvalidatorForOrder(maker, orderHash) -> uint256
  Slot:      5 (verified empirically via the 1inch test)
  Encoding:  raw == 0           -> untouched (filled = 0)
             raw == uint256.max -> fully filled OR cancelled (filled = makingAmount)
             else               -> filled = makingAmount - ~raw

OpenOcean (limit orders unsupported)
  Mainnet does NOT deploy a 1inch-v2-fork LOP. The address in OpenOcean's repo and SDK
  (`0xcC8d695603ce0b43D352891892FcC716c6a7C9f4`) is a custom batch-swap settlement contract
  with no `remainingRaw`/`remaining`/`_remaining` storage exposed. Fill state lives in
  OpenOcean's off-chain orderbook API, not on-chain. We dropped LO support entirely; the
  adapter is atomic-only.
